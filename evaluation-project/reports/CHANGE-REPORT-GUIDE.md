# Change Report Guide

This document defines the structure and requirements for auditable change reports in this HAMR project. These reports document changes that span requirements, models, and code, with full traceability for certification auditing.

## When to Generate a Change Report

Generate a change report after completing any change that modifies artifacts across multiple levels of the development hierarchy (requirements, models, code). This includes:

- Renaming or restructuring data types, requirements, or components
- Adding or modifying requirements and propagating to models and code
- Adding or modifying GUMBO contracts
- Architectural changes (adding/removing components, ports, connections)
- Any change that an auditor would need to trace through the system

## Report File Naming Convention

Reports are stored in `evaluation-project/reports/` with the naming pattern:

```
CR-<NNN>-<short-description>.md
```

Where `<NNN>` is a sequential change report number (e.g., `CR-001`, `CR-002`).

## Required Report Sections

### Header Table

Every report begins with a summary table:

| Field | Description |
|-------|-------------|
| **Change ID** | The CR number |
| **Date** | Date the change was completed |
| **Summary** | One-line description of the change |
| **Scope** | Which artifact levels were affected (Requirements, Models, Code) |
| **Verification** | Summary of test results confirming the change |

### Section 1: Requirements Changes

Document all changes to `requirements/requirements.md`:
- Old vs. new requirement IDs (table format)
- Old vs. new requirement text
- Any new or removed requirements

### Section 2: Model Changes

Document all changes to SysMLv2 `.sysml` files:
- Data type changes (enum values, struct fields)
- Component specification changes (ports, GUMBO contracts)
- System architecture changes (connections, allocations)
- For each change, indicate the **commit hash** and **model validation result** (HAMR type checker outcome)
- Each model change should trace back to the requirement(s) it implements

### Section 3: Code Changes

Document all changes to files under `hamr/microkit/`. Separate into two subsections:

#### 3.1 Auto-Generated Files

Files updated by running HAMR code generation (`sireum_hamr_sysml_codegen`). These are marked with `// Do not edit this file as it will be overwritten if codegen is rerun`. For each file, list:
- File path
- What changed
- Which model element drove the change

#### 3.2 Developer-Written Files

Files marked `// This file will not be overwritten if codegen is rerun` that required manual updates. Separate into:

- **Component implementations** (`*_app.rs`) -- changes to application logic
- **Test files** (`tests.rs`) -- changes to test functions, assertions, strategies

For each file, list the specific changes and trace them to requirements and/or model elements.

### Section 4: Traceability Matrices

Provide the following traceability tables:

#### 4.1 Requirements to Model

| Requirement | Model Element(s) | Model File |
|-------------|-------------------|------------|

Maps each requirement to the SysMLv2 model elements that formalize or implement it.

#### 4.2 Model to Code (Auto-Generated)

| Model Element | Generated Code Artifact(s) |
|---------------|----------------------------|

Maps model elements to the auto-generated code files they produce.

#### 4.3 Requirements to Code (Developer-Written)

| Requirement | Code Implementation | Code File |
|-------------|--------------------|----|

Maps each requirement to the specific lines of developer-written code that implement it.

#### 4.4 Requirements to Tests

| Requirement | Test Function(s) | Test File |
|-------------|-------------------|----|

Maps each requirement to the test functions that verify it. Include:
- Manual unit tests
- Manual GUMBOX (contract-based) tests
- Automated PropTest (property-based) tests

### Section 5: Change Process Summary

A table documenting each step taken:

| Step | Action | Tool/Method | Artifact |
|------|--------|-------------|----------|

Include: requirements edits, model edits, type checking, code generation, developer code edits, test execution.

End with a commit history table listing all commits that comprise the change.

## Formatting Guidelines

- Use tables for structured data (all traceability information should be in tables)
- Use relative file paths from the project root
- Include commit hashes for each artifact-level change
- Include tool invocation results (type checker, test runner) with pass/fail counts
- Keep the report self-contained: an auditor should be able to review the report without needing to read the git diff
