# Testing Guide

How to test HAMR thread components using the generated test infrastructure.

## Overview

HAMR generates test infrastructure for each thread component that supports three styles of testing:

1. **Manual unit tests** -- Direct port manipulation and assertions
2. **Manual GUMBOX tests** -- Contract-based harness that automatically checks GUMBO pre/postconditions
3. **Automated (PropTest) GUMBOX tests** -- Property-based testing with random input generation

The test styles above are listed in order of their level of automation:
- manual unit tests require more effort from the user -- the input test vector must be explicitly design and conditions on expected results must be explicitly coded
- manual GUMBOX tests require the user to explicitly design the input test vector, but test success criteria is determined by HAMR-generated oracles composed of boolean methods (GUMBOX methods) that encode the semantics of model-level GUMBO contracts for the component.  Thus, the user does NOT have to explicitly check for expected results.  The user only has to write GUMBO specifications for the GUMBO.  Checks on expected outputs are produced automatically from the auto-generated GUMBOX methods.
- Automated GUMBOX tests use random value generators generated by HAMR to produce inputs and the GUMBOX-tests harnesses to check for expected results.  Thus, the user does not need to explicitly write input test vectors, nor code that checks for expected results.  

Test automation such as the GUMBOX oracles are automatically derived from the model-level GUMBO contracts.  A potential pitfall is that GUMBO contracts are..
  - incorrect - they incorrectly specify the functional behavior of the component
  - incomplete - some functionality desired for the component may not be reflected in the contract.
Thus, even when GUMBOX capabilities are used, it is still valuable for users or agents to write manual tests and manual GUMBOX text to aid in validation of GUMBO contracts.  Such tests provide scenarios / examples that the developer can write or study to easily understand use cases and intent of component functionality.  

By default, tests are written in `crates/<component>/src/test/tests.rs`, though the user can re-organize the tests into different files.

## Running Tests

Run tests from each component's crate directory:

```bash
cargo test                    # run all tests
cargo test test_name          # run a specific test
cargo test -- --nocapture     # show println/log output
```

Alternatively, use the crate Makefile:

```bash
make test
```

Tests compile and run **without seL4 libraries**. In test mode, the infrastructure uses `Mutex`-guarded statics instead of shared memory, so no `--features sel4` flag is needed.

## Important: `#[serial]` Is Required

All tests **must** use the `#[serial]` attribute from the `serial_test` crate. This is because component state and port APIs use static mutable state that would cause race conditions if tests ran in parallel:

```rust
use serial_test::serial;

#[test]
#[serial]  // REQUIRED
fn my_test() {
    // ...
}
```

## Test Infrastructure Files

These files are auto-generated by HAMR (do not edit):

| File | Purpose |
|------|---------|
| `test/util/test_apis.rs` | `put_<port>()` and `get_<port>()` functions for setting inputs and reading outputs. Also defines `PreStateContainer` structs. |
| `test/util/cb_apis.rs` | Contract-based test harness: `testInitializeCB()`, `testComputeCB()`, `testComputeCBwGSV()`. PropTest macros. |
| `test/util/generators.rs` | PropTest value generators for each data type (`_default()` and `_cust()` strategies). |
| `bridge/<component>_GUMBOX.rs` | Executable GUMBO contracts as boolean functions, used by the test harness. |

## Test Type 1: Manual Unit Tests

Directly construct inputs, run the component, and assert on outputs -- illustrated below using the SNG system.

```rust
mod tests {
    use serial_test::serial;
    use crate::test::util::*;
    use data::*;

    #[test]
    #[serial]
    fn test_initialization() {
        crate::gate_gate_initialize();

        // After initialization, output port should have no value
        // (EventDataPort does not require initialization)
        let output = test_apis::get_output();
        assert!(output.is_none());
    }

    #[test]
    #[serial]
    fn test_drop_top_secret() {
        crate::gate_gate_initialize();
        let msg = SNG_Data_Model::Message {
            security_level: SNG_Data_Model::SecurityLevel::TopSecret,
            payload: 42,
        };
        test_apis::put_input(Some(msg));
        crate::gate_gate_timeTriggered();
        let output = test_apis::get_output();
        assert!(output.is_none(), "TopSecret message should be dropped");
    }

    #[test]
    #[serial]
    fn test_pass_public() {
        crate::gate_gate_initialize();
        let msg = SNG_Data_Model::Message {
            security_level: SNG_Data_Model::SecurityLevel::Public,
            payload: 42,
        };
        test_apis::put_input(Some(msg));
        crate::gate_gate_timeTriggered();
        let output = test_apis::get_output();
        assert!(output == Some(msg), "Public message should pass through");
    }
}
```

### Helper Function Pattern

Reduce boilerplate by defining a helper that initializes, sets input, runs compute, and returns output:

```rust
fn run_gate(input: Option<SNG_Data_Model::Message>) -> Option<SNG_Data_Model::Message> {
    crate::gate_gate_initialize();
    test_apis::put_input(input);
    crate::gate_gate_timeTriggered();
    test_apis::get_output()
}

#[test]
#[serial]
fn test_pass_secret() {
    let msg = SNG_Data_Model::Message {
        security_level: SNG_Data_Model::SecurityLevel::Secret,
        payload: 50,
    };
    let output = run_gate(Some(msg));
    assert!(output == Some(msg));
}
```

### Entry Point Function Names

The initialize and compute functions are exposed at the crate root with the naming convention:

- `crate::<component>_initialize()` -- Calls the initialize entry point
- `crate::<component>_timeTriggered()` -- Calls the compute entry point (periodic)

For example, component `gate_gate` has `crate::gate_gate_initialize()` and `crate::gate_gate_timeTriggered()`.

## Test Type 2: Manual GUMBOX (Contract-Based) Tests

Use `cb_apis::testComputeCB()` to automatically check GUMBO pre/postconditions against the component's actual behavior. This function:

1. Checks GUMBO preconditions on the input
2. Sets up the input ports
3. Runs the compute entry point
4. Reads output ports
5. Checks GUMBO postconditions on the output

```rust
mod GUMBOX_manual_tests {
    use serial_test::serial;
    use crate::test::util::*;
    use data::*;

    #[test]
    #[serial]
    fn test_GUMBOX_top_secret_dropped() {
        let input = Some(SNG_Data_Model::Message {
            security_level: SNG_Data_Model::SecurityLevel::TopSecret,
            payload: 99,
        });
        let result = cb_apis::testComputeCB(input);
        assert!(matches!(result, cb_apis::HarnessResult::Passed));
    }

    #[test]
    #[serial]
    fn test_GUMBOX_public_passed() {
        let input = Some(SNG_Data_Model::Message {
            security_level: SNG_Data_Model::SecurityLevel::Public,
            payload: 42,
        });
        let result = cb_apis::testComputeCB(input);
        assert!(matches!(result, cb_apis::HarnessResult::Passed));
    }

    #[test]
    #[serial]
    fn test_GUMBOX_no_input() {
        let result = cb_apis::testComputeCB(None);
        assert!(matches!(result, cb_apis::HarnessResult::Passed));
    }
}
```

### HarnessResult Outcomes

| Result | Meaning |
|--------|---------|
| `HarnessResult::Passed` | Preconditions met and postconditions satisfied |
| `HarnessResult::RejectedPrecondition` | Input did not satisfy GUMBO preconditions (test skipped) |
| `HarnessResult::FailedPostcondition` | Postconditions violated -- indicates a bug in the component |

### Boundary Value Sweeps

Use loops to test multiple boundary values with the GUMBOX harness:

```rust
#[test]
#[serial]
fn test_GUMBOX_boundary_payloads() {
    let payloads = [i32::MIN, -1, 0, 1, 100, 101, i32::MAX];
    let levels = [
        SNG_Data_Model::SecurityLevel::Public,
        SNG_Data_Model::SecurityLevel::Secret,
        SNG_Data_Model::SecurityLevel::TopSecret,
    ];
    for level in levels {
        for payload in payloads {
            let input = Some(SNG_Data_Model::Message {
                security_level: level,
                payload,
            });
            let result = cb_apis::testComputeCB(input);
            assert!(matches!(result, cb_apis::HarnessResult::Passed),
                "Failed for security_level={:?}, payload={}", level, payload);
        }
    }
}
```

### testComputeCBwGSV

Use `testComputeCBwGSV` when GUMBO contracts reference component state variables (GSV). It takes both port inputs and state variable values.

## Test Type 3: Automated PropTest (Property-Based) Tests

Use HAMR-generated macros with PropTest to automatically generate random inputs and verify GUMBO contracts hold:

```rust
mod GUMBOX_tests {
    use serial_test::serial;
    use proptest::prelude::*;
    use crate::test::util::*;
    use crate::testInitializeCB_macro;
    use crate::testComputeCB_macro;

    const numValidComputeTestCases: u32 = 100;
    const computeRejectRatio: u32 = 5;
    const verbosity: u32 = 2;

    // Test initialization entry point
    testInitializeCB_macro! {
        prop_testInitializeCB_macro,
        config: ProptestConfig {
            cases: numValidComputeTestCases,
            max_global_rejects: numValidComputeTestCases * computeRejectRatio,
            verbose: verbosity,
            ..ProptestConfig::default()
        }
    }

    // Test compute with default random strategy
    testComputeCB_macro! {
        prop_testComputeCB_macro,
        config: ProptestConfig {
            cases: numValidComputeTestCases,
            max_global_rejects: numValidComputeTestCases * computeRejectRatio,
            verbose: verbosity,
            ..ProptestConfig::default()
        },
        api_input: generators::option_strategy_default(
            generators::SNG_Data_Model_Message_strategy_default()
        )
    }
}
```

### PropTest Configuration

| Field | Purpose |
|-------|---------|
| `cases` | Number of test cases to generate |
| `max_global_rejects` | Maximum inputs rejected by preconditions before failing |
| `verbose` | Verbosity level (0=quiet, 2=detailed) |

### Available Macros

| Macro | Purpose |
|-------|---------|
| `testInitializeCB_macro!` | Tests the initialize entry point |
| `testComputeCB_macro!` | Tests the compute entry point (port inputs only) |
| `testComputeCBwGSV_macro!` | Tests the compute entry point (port inputs + state variables) |

## Custom PropTest Strategies

HAMR generates default and customizable strategies in `generators.rs`.

### Default Strategies

Default strategies generate uniformly random values across the full range:

```rust
// Full-range random Message
generators::SNG_Data_Model_Message_strategy_default()

// Wrap in Option (50/50 Some/None)
generators::option_strategy_default(
    generators::SNG_Data_Model_Message_strategy_default()
)
```

### Biased Enum Generation

Weight enum variants to focus testing on specific cases:

```rust
// Heavily bias toward TopSecret (5x weight vs Public and Secret)
generators::SNG_Data_Model_SecurityLevel_strategy_cust(
    1,  // Public weight
    1,  // Secret weight
    5   // TopSecret weight
)
```

Set a weight to 0 to exclude a variant entirely:

```rust
// Only Secret messages (no Public, no TopSecret)
generators::SNG_Data_Model_SecurityLevel_strategy_cust(0, 1, 0)
```

### Custom Field Ranges

Constrain numeric fields to specific ranges:

```rust
// Messages with payload in [-5, 105] and Secret-only security level
generators::SNG_Data_Model_Message_strategy_cust(
    (-5i32..=105i32),  // payload range
    generators::SNG_Data_Model_SecurityLevel_strategy_cust(0, 1, 0)
)
```

### Option Bias (Some vs None Ratio)

Control how often `Some` vs `None` is generated:

```rust
// 5:1 ratio of Some to None
generators::option_strategy_bias(5, base_strategy)

// Always Some, never None
base_strategy.prop_map(Some)
```

### Complete Custom Strategy Example

Combine multiple customizations:

```rust
testComputeCB_macro! {
    prop_testComputeCB_TopSecret_biased,
    config: ProptestConfig {
        cases: 100,
        max_global_rejects: 500,
        verbose: 2,
        ..ProptestConfig::default()
    },
    api_input: generators::option_strategy_bias(
        5,  // bias toward Some (5:1 vs None)
        generators::SNG_Data_Model_Message_strategy_cust(
            any::<i32>(),  // full i32 range
            generators::SNG_Data_Model_SecurityLevel_strategy_cust(
                1,  // Public
                1,  // Secret
                5   // TopSecret (heavily biased)
            )
        )
    )
}
```

## Test Design Strategies

### Boundary Value Analysis

Test at, just below, and just above boundaries:

```rust
// For [0,100] clamping:
let payloads = [-1, 0, 1, 99, 100, 101];
```

### Extreme Values

Include type limits:

```rust
let payloads = [i32::MIN, -1, 0, 1, 100, 101, i32::MAX];
```

### Precondition Verification

Confirm GUMBO `assume` constraints reject invalid inputs:

```rust
// Filter has assume: input.security_level != TopSecret
// Sending TopSecret should be rejected by precondition
let result = cb_apis::testComputeCB(Some(top_secret_msg));
assert!(matches!(result, cb_apis::HarnessResult::RejectedPrecondition));
```

### Integration Constraint Verification

Confirm GUMBO `guarantee` constraints hold across all inputs:

```rust
// Gate has guarantee: output.security_level != TopSecret
// Verify for all security levels
for level in [Public, Secret, TopSecret] {
    let output = run_gate(Some(Message { security_level: level, payload: 42 }));
    if let Some(out_msg) = output {
        assert!(out_msg.security_level != SecurityLevel::TopSecret);
    }
}
```

### Combining Test Types

Use all three test types together for thorough coverage:

1. **Manual unit tests** for specific requirements and edge cases
2. **Manual GUMBOX tests** for boundary sweeps with automatic contract checking
3. **PropTest** for broad random exploration with custom strategies targeting high-risk areas

## Generator Naming Convention

HAMR generates strategy functions following this pattern:

```
generators::<Package>_<Type>_strategy_default()   // full-range random
generators::<Package>_<Type>_strategy_cust(...)   // customizable parameters
```

For example:
- `generators::SNG_Data_Model_Message_strategy_default()`
- `generators::SNG_Data_Model_Message_strategy_cust(payload_range, security_level_strategy)`
- `generators::SNG_Data_Model_SecurityLevel_strategy_default()`
- `generators::SNG_Data_Model_SecurityLevel_strategy_cust(pub_weight, sec_weight, ts_weight)`
